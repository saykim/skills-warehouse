# 목표-실행 일치도 체크 프레임워크

## 핵심 질문 3가지
1. **"지금 뭐하고 있나?"** - 현재 활동 파악
2. **"원래 뭐하기로 했나?"** - 계획된 활동 확인
3. **"왜 다른 걸 하고 있나?"** - 갭(Gap) 분석

---

## Phase 1: 목표 정렬 (Goal Alignment)

### 1.1 목표 명확성 체크
```
원래 목표: [          ]
↓
현재 작업: [          ]
↓
정렬도: ☐ 완전일치  ☐ 부분일치  ☐ 불일치
```

**체크포인트:**
- [ ] 최상위 목표가 명확히 정의되어 있는가?
- [ ] 현재 작업이 그 목표에 직접 기여하는가?
- [ ] 우선순위가 변경되었다면 왜? 승인은?

### 1.2 목표 분해 검증
```
최상위 목표
├─ Sub-Goal 1
│  ├─ Task 1-1
│  └─ Task 1-2
└─ Sub-Goal 2
   └─ Task 2-1
```

**확인 사항:**
- [ ] 모든 Sub-Goal이 상위 목표에 기여하는가?
- [ ] Sub-Goal이 MECE하게 분해되었는가?
- [ ] 각 Task가 해당 Sub-Goal 달성에 필수인가?

## Phase 2: 계획 vs 실행 갭 분석

### 2.1 일정 추적 (Schedule Tracking)

| 마일스톤 | 계획 날짜 | 실제 날짜 | 상태 | 지연 원인 | 대응 |
|---------|----------|----------|------|----------|------|
| M1      |          |          | ✅/⚠️/❌ |          |      |
| M2      |          |          | ✅/⚠️/❌ |          |      |

**분석 질문:**
- 지연된 마일스톤의 Critical Path 영향도는?
- 후속 마일스톤에 연쇄 지연 리스크는?
- 일정 회복 가능한가? 어떻게?

### 2.2 범위 추적 (Scope Tracking)

| 기능/작업 | 계획 | 실제 | 변경 사유 | 승인 여부 |
|----------|------|------|----------|----------|
|          | In   | In   |          | ✅       |
|          | In   | Out  |          | ⚠️       |
|          | Out  | In   |          | ❌       |

**위험 신호:**
- 🚨 **Scope Creep**: Out → In (승인 없이)
- 🚨 **Silent Cut**: In → Out (문서화 없이)
- 🚨 **Priority Shift**: P0 → P2로 강등 (근거 없이)

### 2.3 리소스 추적 (Resource Tracking)

```
계획된 리소스:
- 개발자 3명 × 2개월 = 6 man-months
- 디자이너 1명 × 1개월 = 1 man-month

실제 투입:
- 개발자 2명 × 2.5개월 = 5 man-months (부족)
- 디자이너 1명 × 0.5개월 = 0.5 man-month (부족)
```

**체크포인트:**
- [ ] 계획 대비 실제 리소스 투입량?
- [ ] 부족분을 어떻게 메울 것인가?
- [ ] 리소스 부족이 품질/일정에 미친 영향?

## Phase 3: 진행률 vs 소모율 분석

### 3.1 번다운 차트 분석 (Burndown Analysis)

```
이상적 진행:   [########........] 50% 완료, 50% 남음
실제 진행:     [#####...........] 30% 완료, 70% 남음
소모된 시간:   [##########......] 60% 소모, 40% 남음

🚨 위험: 30% 진행했는데 60% 시간 소모 → 예상 완료율 50%
```

**즉시 답해야 할 질문:**
1. 왜 예상보다 느린가? (기술적 어려움? 리소스 부족?)
2. 나머지 40% 시간으로 70% 작업 가능한가?
3. 범위 조정? 일정 연장? 리소스 추가?

### 3.2 작업 효율성 추적

| 주차 | 계획 Story Point | 실제 완료 SP | Velocity | 트렌드 |
|-----|-----------------|-------------|----------|--------|
| W1  | 20              | 15          | 75%      | ↓      |
| W2  | 20              | 18          | 90%      | ↑      |
| W3  | 20              | 12          | 60%      | ↓↓     |

**패턴 분석:**
- Velocity 하락 트렌드? → 팀 번아웃? 기술 부채?
- Velocity 안정? → 계획이 현실적
- Velocity 상승? → 초기 과소평가 또는 팀 적응

## Phase 4: 리스크 및 Blocker 추적

### 4.1 리스크 현황

| 리스크 | 식별 시점 | 발생 확률 | 영향도 | 대응 상태 | 실제 발생 |
|--------|----------|----------|--------|----------|----------|
| R1     |          | H/M/L    | H/M/L  | ✅/⚠️/❌  | Yes/No   |

**위험 신호:**
- 🚨 High-Impact 리스크 대응 없음
- 🚨 예상 리스크가 실제 발생했는데 대응 안 함
- 🚨 새로운 리스크 식별 안 됨 (리스크 관리 부재)

### 4.2 Blocker 추적

```
Blocker: [외부 API 연동 대기]
├─ 발생 시점: Week 2
├─ 차단된 작업: 결제 기능 개발 (P0)
├─ 예상 해결: Week 4 (2주 지연)
├─ 대응 현황: API 팀과 매일 Sync-up
└─ 대안 검토: Mock API로 선개발?
```

**체크포인트:**
- [ ] 모든 Blocker가 추적되고 있는가?
- [ ] 각 Blocker에 대한 Escalation 계획은?
- [ ] 대안 (Workaround) 검토했는가?

## Phase 5: 품질 vs 속도 트레이드오프

### 5.1 기술 부채 축적

```
Sprint 1: 빠른 개발 → 테스트 커버리지 30%
Sprint 2: 빠른 개발 → 리팩토링 미루기
Sprint 3: 빠른 개발 → 버그 누적 20개
...
Sprint 10: 🚨 속도 70% 감소 (기술 부채 상환)
```

**경고 신호:**
- 버그 누적 증가율 > 해결률
- 코드 리뷰 승인 시간 증가
- "나중에 고치자"가 문서화 안 됨

### 5.2 품질 메트릭 추적

| 지표 | 목표 | 현재 | 트렌드 | 조치 필요 |
|-----|------|------|--------|----------|
| 테스트 커버리지 | 80% | 65% | ↓ | ⚠️ |
| 버그 밀도 | <5/KLOC | 8/KLOC | ↑ | 🚨 |
| Code Review 시간 | <1일 | 3일 | ↑ | ⚠️ |

**즉시 대응 필요:**
- 🚨 품질 메트릭이 목표 대비 20% 이상 악화
- 🚨 트렌드가 2주 이상 지속 악화

## Phase 6: 커뮤니케이션 및 투명성

### 6.1 Status Update 체크

```
계획된 보고:   주간 리포트 (매주 금요일)
실제 보고:     [✅] W1  [✅] W2  [❌] W3  [⚠️] W4 (부실)
```

**문제 패턴:**
- 나쁜 소식은 늦게, 좋은 소식은 빨리
- 문제를 "해결 중"이라고만 하고 진척 없음
- Stakeholder가 다른 경로로 문제 파악

### 6.2 의사결정 투명성

| 결정 사항 | 결정 시점 | 결정권자 | 문서화 | 팀 공유 |
|----------|----------|----------|--------|---------|
| 범위 축소 | W3 | PM | ✅ | ❌ |

**위험 신호:**
- 주요 결정이 문서화 안 됨
- 팀이 결정 사항 모름
- "왜 이렇게 했죠?" → 답 없음

## Phase 7: 냉정한 현실 평가

### 7.1 출시 가능성 판단

```
목표 출시일: 2주 후
현재 진행률: 60%
남은 작업: 40% → 최소 3주 필요

결론: 🚨 일정 준수 불가능
```

**3가지 선택:**
1. **범위 축소**: 40% → 25% (MVP만)
2. **일정 연장**: 2주 → 4주
3. **리소스 추가**: +2명 투입 (품질 위험)

**냉정한 질문:**
- "정말 2주 안에 가능하다고 생각하나?"
- "무리하면 뭘 잃나?" (품질? 팀 번아웃?)
- "연기하면 뭘 잃나?" (시장 기회? 신뢰?)

### 7.2 Go/No-Go 체크리스트

출시 전 필수 확인:
- [ ] 모든 P0 요구사항 완료
- [ ] Critical Path 버그 0건
- [ ] 성능 테스트 통과
- [ ] 보안 리뷰 완료
- [ ] Rollback 계획 준비
- [ ] 모니터링 대시보드 구축
- [ ] On-call 체계 수립

**하나라도 No면 → 출시 불가 또는 높은 리스크**

## 일반적인 갭(Gap) 패턴

### 패턴 1: Silent Drift (조용한 표류)
```
Week 1: 목표 A에 집중
Week 2: 목표 A + 목표 B (동시에)
Week 3: 목표 B에 집중 (목표 A는?)
Week 4: 목표 C 시작 (A, B는?)
```
→ 우선순위 없이 이것저것 → 아무것도 완료 안 됨

### 패턴 2: Feature Factory (기능 공장)
```
목표: 사용자 유지율 10% 증가
실행: 기능 10개 출시
결과: 유지율 변화 없음
```
→ 기능 만들기에만 집중, 목표 달성은 망각

### 패턴 3: Analysis Paralysis (분석 마비)
```
Week 1-2: 요구사항 분석
Week 3-4: 설계 검토
Week 5-6: 기술 검증
Week 7-8: 프로토타입...
```
→ 완벽을 추구하다 실행 못함

### 패턴 4: Scope Creep (범위 확장)
```
원래: MVP 3개 기능
현재: 10개 기능 + 4개 통합 + 리팩토링
```
→ "이것도 필요하지 않을까?" → 끝없는 확장

### 패턴 5: Tech Debt Explosion (기술 부채 폭발)
```
Sprint 1-5: 빠르게 개발 (테스트 생략)
Sprint 6-10: 버그 수정에 80% 시간 소모
```
→ 단기 속도를 위해 장기 속도 희생

## PM으로서 해야 할 냉정한 질문

### 계획 단계
1. "이 목표, 정말 달성 가능한가?"
2. "최악의 시나리오는?"
3. "누가, 언제까지, 무엇을 하나?"

### 실행 단계
1. "지금 하는 일이 목표와 직접 연결되나?"
2. "이 속도로 가면 목표 달성 가능한가?"
3. "문제를 숨기고 있진 않나?"

### 검토 단계
1. "우리가 놓친 리스크는?"
2. "Stakeholder는 현실을 알고 있나?"
3. "출시해도 되나? 정말?"

✅ 목표가 명확하고 팀이 다 아는가?
✅ 진행률과 소모율이 정렬되는가?
✅ Blocker가 추적/관리되는가?
✅ 나쁜 소식이 투명하게 공유되는가?
✅ 출시 기준이 명확하고 지켜지는가?

→ 하나라도 ❌면 즉시 조치 필요
